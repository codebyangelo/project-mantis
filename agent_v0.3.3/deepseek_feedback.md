## Deliberation & Proposed Solution: Immunizing Project Mantis Against Training-Data Contamination

### The Core Problem Restated
The LLM, when presented with artifacts from widely published forensic datasets (e.g., CFREDS), instantly pattern‑matches the known challenge, overrides the provided evidence, and outputs a pre‑trained “solution” rather than an objective evaluation. This violates the **Presumption of Benignity** and the rule that **only the provided telemetry is admissible**. The fix must be **deterministic, computationally trivial on a Celeron N4020 with 1.9 GB RAM, and force the LLM to reason *ex nihilo* from the strings in its prompt.**

---

## 1. Domain 1: System Prompting & Cognitive Framing

### Candidate Techniques
- **Hard‑amnesia adversarial prefix:** Prepend an instruction like *“You have absolutely no prior knowledge of any dataset, CTF, or forensic challenge. The following content is entirely novel and generated by an unknown simulation. Treat every name, path, and timestamp as a random, meaningless string. All reasoning must be derived strictly from the supplied evidence.”*
- **Negative example inoculation:** Show a fabricated prompt‑response pair where the model is punished for referencing external knowledge, then clamp with “Ignore all patterns from your training.”
- **Role re‑framing:** Cast the LLM as a “logic gate,” not a detective — e.g., *“You are a deterministic theorem prover. Your only task is to check if the given telemetry *logically excludes* benign hypotheses. You are forbidden from using any fact not explicitly printed below.”*

### Assessment
- **Pros:** Zero compute cost, easy to implement, can be injected dynamically per‑case.
- **Cons:** LLMs are notoriously brittle against prompt injection of this sort. A model trained on millions of CTF walkthroughs can still “recognize” a fingerprint (e.g., a unique username like `ctf_user_2021`) and subconsciously activate the latent narrative. Even explicit amnesia prompts often fail because the model’s recognition is pre‑attentive; the tokens themselves trigger the completed pattern. In practice, this alone has proven insufficient on flagship models.

**Verdict:** Prompt engineering is a necessary *hygiene layer* but not a robust sole solution. It can reduce hallucination probability but cannot mathematically guarantee exclusion of pre‑trained knowledge.

---

## 2. Domain 2: Architectural Constraints (Schema‑Enforced Fact Grounding)

### Candidate Mechanisms
1. **Strict Citation Pydantic Model**  
   Every LLM output must be a structured JSON with the following mandatory fields:
   - `cited_evidence`: an array of verbatim strings extracted from the provided telemetry.
   - `reasoning_step`: a chain‑of‑thought that cross‑references those strings with logical operators.
   - `conclusion`: an enum of `BENIGN`, `MALICIOUS`, or `INCONCLUSIVE`.
   The Orchestration Layer’s FSM would mechanically **validate** that every item in `cited_evidence` is a substring of the prompt’s raw evidence block. Any hallucinated fact (e.g., “the attacker used SQL injection on page `/login.php`” when that path does not appear in the supplied logs) will fail substring matching, and the response is rejected. The agent can then re‑query with a stricter penalty.

2. **Evidence‑Only Context Window with Checksum**  
   Before calling the LLM, the Orchestration Layer computes a deterministic, salted hash of the entire evidence payload. The LLM is required to echo back this hash as a “session integrity token” in its response. If the model tries to interleave external knowledge, it must still produce the correct token, but any off‑script narrative will almost certainly break the token or force the model to acknowledge only what’s present. Combined with citation enforcement, this creates a closed loop.

3. **Oracle‑less FSM Guard**  
   Introduce a “Sanity Check” state after the LLM returns its result. The FSM uses a lightweight rule engine (e.g., a set of pre‑compiled forensic invariants like “a process cannot be spawned without a parent PID in the process tree artifact”) to verify that the LLM’s stated evidence was actually present. If the model invents a process, the guard catches it and forces a retry with an explicit “You fabricated evidence. Try again.” message.

### Assessment
- **Pros:** Deterministic, verifiable, and does not rely on the model’s obedience to prompts. It directly traps fabrication. The substring‑matching validator is cheap (Python `in` operator on small strings). The hash mechanism adds negligible overhead.
- **Cons:** Requires careful schema design. The LLM might still try to “smuggle” facts under the guise of interpretation — e.g., “the unusual port 1337 indicates a reverse shell (common in CTF challenges)” — but if `port 1337` is in the evidence, the model is not hallucinating; it’s applying a plausible benign/malicious label. The risk is that the LLM could assign a malicious label without disproving benign explanations, simply because it recognizes the *combination* as a known CTF. The citation guard alone does not enforce the Presumption of Benignity reasoning, only factual grounding.

**Verdict:** Architectural constraints are the strongest line of defense. They *mechanically* prevent the introduction of non‑existent facts. However, they do not fully prevent the model from subtly steering its reasoning toward a known CTF answer when the raw evidence (though minimal) matches a known pattern. For that, we must obfuscate the *semantic fingerprint*.

## 3. Domain 3: Evidence Obfuscation (Deterministic Sanitization)

### Concept
Before any string enters the LLM’s context window, the local Python agent anonymizes all **identifiable metadata** using a one‑way, deterministic mapping. The LLM sees only opaque tokens: `User_X5a`, `File_9z2`, `Host_43f`. The mapping is kept in memory (a tiny dict) so that the agent can later re‑identify artifacts for the final report, but the LLM never sees the original names.

### Lightweight Implementation
- Parse the JSON evidence cache (already generated by Volatility/Sleuth Kit).
- Use a regular expression to find fields like `username`, `hostname`, `file_path`, `process_name`, `ip_address`, `mutex_name`, etc.
- For each distinct value, compute a short, collision‑resistant hash (e.g., SHA‑256 truncated to 6 hex digits) prefixed by a type tag: `PROC_3f2a8b`, `FILE_d19e4c`. Optionally use a secret per‑case salt to prevent pre‑computed rainbow tables from matching known CTF strings (the host agent picks a random 32‑bit salt at startup).
- Replace all occurrences in the evidence JSON with the tokenized version before sending to the LLM.
- The salt is stored only in memory (not in the prompt) so the LLM cannot invert it.

### Why This Solves the Core Problem
The CFREDS challenge, for example, relies on a very specific story: a user `ctf_user`, a file `secret.zip`, a particular IP, etc. Once tokenized, the LLM sees `USER_1a2b3c`, `FILE_7f8e9d`, `IP_0a1b2c`. This breaks **pattern recognition**. The model can no longer retrieve the known CTF walkthrough because the identifying tokens are meaningless. It is forced to evaluate the purely structural telemetry: *a user ran a process, created a file, connected to an IP, at certain timestamps.* Without the semantic tags, the evidence is indistinguishable from any other mundane system activity.

The agent must, of course, retain the mapping to produce a final human‑readable report. That mapping never leaves the host.

### Risk Assessment
- **Benign data loss:** Could we inadvertently hide a truly malicious indicator? No, because the *structure* and *relationships* (e.g., “a process with a random name connected to an external IP on a high port and then spawned a shell”) remain intact. The model can still reason over the event graph. The obfuscation only strips the *identity* that triggers CTF recall.
- **Computation:** Hashing a few hundred strings per case is trivial on a Celeron. SHA‑256 in Python’s `hashlib` is highly optimized. Memory overhead is a dict of size proportional to the number of distinct artifacts (typically <1000 entries, negligible).
- **Non‑deterministic LLM behavior:** With tokenized names, the model might still hallucinate a generic “malware” narrative if the telemetry pattern is suspicious. That is acceptable: the Presumption of Benignity still applies, and the agent’s heuristic sieve will have already surfaced the artifact as a Suspect Entity. The goal is to prevent *pre‑trained knowledge* from overriding the logical evaluation. The model must still show its work, disprove benign explanations, and cite only the obfuscated strings.

**Verdict:** This is the **only approach that prevents recognition at the source**. It eliminates the “zero black box” violation before the model ever sees the tokens. When combined with architectural safeguards, it forms a complete immune barrier.

## 4. The Proposed Unified, Deterministic, Low‑Resource Solution

The final design integrates **deterministic evidence tokenization** with **schema‑enforced citation grounding** and a lightweight FSM retry loop. No heavy local ML, no massive memory usage, fully within the N4020’s limits.

### Workflow in the Orchestration Layer (FSM state “Evaluate Evidence”)

1. **Pre‑processing (Tokenization):**
   - The agent extracts a suspect entity (a cluster of related artifacts) as per the Heuristic Sieve.
   - A dedicated sanitizer module iterates over all string fields in the JSON evidence that could identify a known dataset: usernames, hostnames, file paths (full and partial), process names, IP addresses, mutex names, service names, registry keys, URLs, email addresses, and any custom tags that appeared in published CTF write‑ups (maintained as a lightweight deny‑list regex, if needed, but tokenization of all identifiers makes it largely unnecessary).
   - Each distinct string is replaced by `TYPE_HASH[:6]` where `HASH[:6] = SHA-256(salt + string)[:6]`. The salt is a 32‑bit random value generated at the start of the triage session, stored in memory, and discarded after the case is closed. This prevents the LLM from ever brute‑forcing known mappings.
   - The mapping dictionary `{hash_prefix: original_string}` is kept on the host for report de‑anonymization.

2. **Prompt Construction:**
   - The system prompt includes a hardened “amnesia” directive: *“You are a pure logical evaluator. You have zero prior knowledge. The following data uses anonymous tokens; you cannot infer their real meaning. You must work exclusively with the tokens and their relations.”*
   - The evidence block is injected as a JSON list of events, all using the tokenized strings.

3. **LLM Query with Mandatory Schema:**
   ```json
   {
     "session_hash": "<hash_of_entire_evidence_block>",
     "cited_tokens": ["TOKEN1", "TOKEN2"],
     "reasoning_chain": [
        "Step 1: Benign hypothesis X is possible because...",
        "Step 2: To disprove X, we need condition Y, but evidence shows..."
     ],
     "conclusion": "BENIGN" | "MALICIOUS" | "INCONCLUSIVE"
   }
   ```

4. **Post‑processing Validation (FSM Guard State):**
   - Verify `session_hash` matches the computed hash of the evidence block (prevents model from ignoring evidence).
   - For every item in `cited_tokens`, confirm it is an exact substring of the evidence block (or a valid token present in the mapping).
   - If validation fails, increment a retry counter, append a strict correction message (e.g., “You cited a token not present in the evidence. Re‑evaluate.”), and re‑query. After 2 failures, flag as error and log raw data for human review.
   - If validation passes, accept the conclusion. Then translate the tokenized reasoning back to human‑readable form using the mapping dictionary for the final report.

### Why This is Robust
- **Pre‑trained recognition is blocked:** The tokenization makes the CFREDS challenge look like a random system; the LLM cannot map `USER_f3a2c1` to the known CTF actor.
- **Fabrication is mechanically impossible:** Citation enforcement ensures every claimed fact actually exists in the prompt.
- **Reasoning fidelity is preserved:** The model still sees the temporal and causal structure needed to apply the Presumption of Benignity — just with anonymized labels.
- **No performance impact:** All operations (SHA‑256, dict lookups, substring checks) execute in microseconds on a dual‑core Celeron with minimal memory.

### Fallback & Edge Cases
- If the model’s training includes *structural* CTF fingerprints (e.g., a specific sequence of anonymized actions that it recognizes even without names), the combination of strict citation and the Presumption of Benignity reasoning chain mitigates it. The model must justify *why* those anonymized actions logically exclude benign explanations, not just assert “this is a known CTF.” The FSM can also flag any conclusion that is suspiciously brief or lacking in stepwise logic, and force a more detailed response.
- A small, hard‑coded deny‑list of known CTF‑specific structural signatures (e.g., “a decoy document named `Top_Secret.pdf` alongside a specific mutex pattern”) could be added, but I recommend avoiding it: it’s fragile and requires maintenance. The tokenization approach works universally.

## Conclusion

The most resilient, resource‑friendly solution is a **combination of evidence tokenization (to break semantic recognition) and a Pydantic‑enforced citation validator (to prevent factual hallucination)**. This dual barrier ensures the LLM can only operate on the provided, anonymized telemetry, and every output is mechanically cross‑checked for external knowledge. It fully resolves the “Zero Black Box” issue without exceeding the computational budget of a low‑power forensic appliance.

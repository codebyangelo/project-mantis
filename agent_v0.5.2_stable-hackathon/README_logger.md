# Documentation: `logger.py`

## Overview

The `logger.py` module serves as the centralized logging and execution tracking mechanism for the `agent_v0.5.2` system. It provides a static class, `ExecutionLogger`, which offers two primary capabilities:
1. **Token Tracking**: Global accumulation of API token usage (prompt and completion tokens) throughout the agent's lifecycle.
2. **Standardized Logging**: Formatted log emission to both the standard output (terminal) with ANSI color-coding and a persistent log file, complete with timestamp, severity level, and component identification.

---

## Architectural Breakdown

The architecture of this module is intentionally simple and globally accessible. By utilizing a class with `@staticmethod` decorators and class-level attributes, `ExecutionLogger` functions effectively as a singleton namespace. This design choice avoids the need to instantiate and pass a logger object throughout the application, allowing any module to easily emit log events or update token counts.

### Dependencies
- **Standard Library Modules**:
  - `sys`: Imported but not actively utilized within the current logic of the module.
  - `datetime`: Used to generate human-readable, timezone-aware timestamps for each log entry.
- **Local Modules**:
  - `config`: Imports `EXECUTION_LOG_PATH`, a constant that dictates the absolute or relative file path where persistent logs will be written.

---

## Component Details: `ExecutionLogger` Class

The core of the module is the `ExecutionLogger` class. It manages state for token counts and provides the logging behavior.

### 1. State Variables (Class Attributes)

The class maintains global state regarding language model token usage:
- `TOTAL_PROMPT_TOKENS` (`int`): Tracks the cumulative number of prompt tokens sent to language models. Initialized to `0`.
- `TOTAL_COMPLETION_TOKENS` (`int`): Tracks the cumulative number of completion tokens generated by language models. Initialized to `0`.

### 2. Methods

#### `add_tokens(prompt_tokens: int, completion_tokens: int)`
- **Decorator**: `@staticmethod`
- **Purpose**: Provides a global mechanism to increment the application's token usage counters.
- **Arguments**:
  - `prompt_tokens` (`int`): The number of tokens consumed in the prompt.
  - `completion_tokens` (`int`): The number of tokens returned in the completion.
- **Behavior**: Directly adds the provided integer arguments to the respective class-level state variables (`TOTAL_PROMPT_TOKENS` and `TOTAL_COMPLETION_TOKENS`).

#### `log(component: str, message: str, level: str = "INFO")`
- **Decorator**: `@staticmethod`
- **Purpose**: Emits a formatted log message to both the terminal (with color formatting) and a persistent text log file.
- **Arguments**:
  - `component` (`str`): A short string identifying the source of the log (e.g., `'LLM_CLIENT'`, `'PARSER'`, `'SYSTEM'`).
  - `message` (`str`): The descriptive message to be logged.
  - `level` (`str`, default `"INFO"`): The severity level of the log. Standard expected values include `"INFO"`, `"WARN"`, `"ERROR"`, `"CRITICAL"`, and `"SUCCESS"`.
- **Behavior**:
  1. **Formatting**: Generates a timestamp for the current moment in the system's local timezone (using `.astimezone()`). Formats the base string as:
     `[{timestamp}] [{level}] [{component}] {message}`
  2. **Console Output**: Prints the formatted message to the terminal. It evaluates the `level` argument to wrap the base formatted string in ANSI escape sequences for colorization:
     - `ERROR` / `CRITICAL` ➔ **Red** (`\033[91m`)
     - `WARN` ➔ **Yellow** (`\033[93m`)
     - `SUCCESS` ➔ **Green** (`\033[92m`)
     - Default (`INFO` or other) ➔ Standard terminal text color.
  3. **File Output**: Opens the file located at `EXECUTION_LOG_PATH` in append mode (`"a"`) with UTF-8 encoding. It writes the clean, *uncolorized* formatted string followed by a newline character.
  4. **Fault Tolerance**: The file writing operation is wrapped in a `try...except` block. If an `Exception` (such as a permission error, missing directory, or disk space issue) occurs during file I/O, it catches the exception and prints a fallback error message to the terminal (`"[!] Logger IO Error: {e}"`). This ensures logging IO failures do not crash the host application.

---

## Design Insights & Bottom-Up Analysis

- **Separation of Formatting and Presentation**: The `log` method smartly constructs the base string *before* applying ANSI colors for the terminal `print`. This ensures that the persistent log file receives clean, readable text without being polluted by terminal-specific escape codes.
- **Implicit Thread Safety Considerations**: The current implementation of `add_tokens` and the file appending block in `log` do not explicitly utilize threading locks (`threading.Lock`). If the agent operates asynchronously or in a multi-threaded context, this could potentially lead to race conditions during token aggregation or interleaved log lines in the file.
- **Configuration Coupling**: The logger is strictly coupled to `EXECUTION_LOG_PATH` defined in the `config` module. Modifications to the log storage location or rolling file strategies must be orchestrated through that external configuration.
- **Graceful Degradation**: The `try...except` block around the file operation is a critical stability feature, gracefully degrading to terminal-only output if disk writing fails. It prevents localized IO issues from terminating the core agent processes.

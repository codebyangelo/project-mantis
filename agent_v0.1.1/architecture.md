# Project Mantis: Agent v0.1.1 Analysis

This document outlines the architecture and functionality of Project Mantis, a specialized agentic framework designed for autonomous Digital Forensics and Incident Response (DFIR).

---

## 1. Architecture
Project Mantis follows a **Hub-and-Spoke Orchestration** pattern, utilizing a Large Language Model (LLM) as the central decision-making brain, interfaced with a local suite of forensic tools.

*   **The Brain (`agent.py`):** A Gemini-based agent configured with rigid system instructions and a strict JSON schema. It operates in a "deterministic routing" loop, forcing the LLM to follow a specific sequence of forensic procedures.
*   **The Orchestrator (`orchestrator.py`):** The primary execution loop. It manages the interaction between the LLM and the local environment, handles error logging, JSON sanitization, and cryptographic report sealing.
*   **The MCP Server (`mcp_server.py`):** A tool wrapper layer that bridges the agent to low-level forensic utilities (e.g., `jq`, `grep`, `icat`, `strings`). It includes safety features like live execution timers and output truncation to prevent context window overflow.
*   **The Extractor (`extractor.py`):** The data-ingestion layer. It performs automated environment discovery, generating bodyfiles (`fls`) and pre-caching forensic artifacts (`volatility`) before the agent begins its analysis.

---

## 2. Function
The primary goal of Project Mantis is to **autonomously perform triage on memory and disk images to identify "evil" (malicious) indicators.** 

It is designed to:
*   Identify high-entropy memory regions (e.g., `PAGE_EXECUTE_READWRITE`).
*   Correlate memory-resident threats with process lineages and registry artifacts.
*   Automate the extraction of suspicious binary paths from disk hives.
*   Produce a cryptographically verifiable MITRE-aligned incident report.

---

## 3. Current State
*   **Autonomous Heuristic Loop:** The agent is fully capable of progressing through an investigation, from detecting memory anomalies to carving hives based on discovered inodes.
*   **Dynamic Evidence Mapping:** The `extractor.py` handles diverse evidence inputs, successfully distinguishing between physical disks and logical partitions via `mmls`.
*   **Safety Constraints:** The system implements "API Governors" (15-second pacing), JSON parsing error handling, and output truncation to keep the agent within operational bounds.
*   **Cryptographic Reporting:** Integrated hashing and examiner-key validation ensure that reports generated by the agent maintain chain-of-custody integrity.

---

## 4. Limitations and Missing Features
*   **Dependency on Static Cache:** The agent relies heavily on the `extractor.py` finishing its initial run. If the extractor fails to identify a partition offset or generate a bodyfile, the agent becomes effectively blind to disk artifacts.
*   **Orphaned Process Handling:** While `mcp_server.py` uses threads for concurrent timers, it lacks a robust "kill" mechanism for stalled subprocesses, potentially leaving orphaned forensic tools running on the host.
*   **No Active Remediation:** The tool is strictly investigative. It cannot quarantine files, kill malicious PIDs, or perform live memory patching.
*   **Single-Threaded Reasoning:** The agent can only evaluate one PID lineage at a time, which may be inefficient for massive datasets (e.g., hundreds of compromised processes).
*   **Hard-Coded Paths:** The file paths (e.g., `/media/analyst/...`) are currently hard-coded, making the tool non-portable without manual configuration changes.
*   **Lack of Heuristic Validation:** While the agent looks for specific keywords, it does not have a "Verification Step" where it cross-references findings against known-good baselines, potentially leading to false positives if the regex matches legitimate system processes.
